Por que Performance de Queries Importa?

Um aplicação web pode ser perfeita em tudo — código limpo, arquitetura impecável, interface amigável — mas se as queries forem lentas, seus usuários ficarão frustrados. Performance no banco de dados é crucial porque:

  • Escalabilidade: Queries otimizadas suportam muito mais usuários simultâneos
  • Custo operacional: Bancos de dados menos exigidos usam menos recursos (CPU, RAM)
  • Experiência do usuário: Respostas rápidas = aplicações responsivas
  • Tendência exponencial: Conforme seus dados crescem, queries lentas ficam ainda mais lentas

EXPLAIN: Seu Melhor Amigo

Antes de otimizar, você precisa entender como o PostgreSQL executa suas queries. Use o comando EXPLAIN para visualizar o plano de execução: 

EXPLAIN ANALYZE
SELECT u.id, u.name, COUNT(o.id) as order_count
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.created_at > '2025-01-01'
GROUP BY u.id, u.name;

A saída mostra o plano de execução com informações como:

  • Seq Scan: Busca sequencial (lê toda a tabela). Geralmente lenta.
  • Index Scan: Usa um índice. Geralmente rápida.
  • Cost: Número relativo de esforço (quanto maior, pior)
  • Rows: Quantidade estimada vs real de linhas (discrepâncias indicam problema)

Estratégia 1: Índices Inteligentes

Índices são a ferramenta mais poderosa de otimização, mas precisam ser bem planejados. Um índice ruim é pior que nenhum índice.

B-Tree (Padrão)

Use para buscas exatas e range queries: 

-- Índice simples
CREATE INDEX idx_users_email ON users(email);

-- Índice composto (ordem importa!)
CREATE INDEX idx_orders_user_date 
  ON orders(user_id, created_at DESC);

BRIN (Block Range Index)

Perfeito para dados que crescem sequencialmente (timestamps, IDs). Muito mais leve que B-Tree: 

-- Para tabelas enormes com dados sequenciais
CREATE INDEX idx_logs_created_brin 
  ON logs USING BRIN (created_at);

GiST/GIN (Para Buscas Complexas)

Use para buscas full-text, arrays e dados geoespaciais: 

-- Full-text search
CREATE INDEX idx_articles_content 
  ON articles USING GIN(to_tsvector('portuguese', content));

-- Array search
CREATE INDEX idx_tags_array 
  ON posts USING GIN(tags);

Estratégia 2: Evitar Seq Scan Desnecessário

Às vezes o PostgreSQL escolhe fazer uma busca sequencial mesmo quando há um índice disponível. Isso ocorre quando:

  • A query espera retornar muitas linhas (>5% da tabela)
  • A coluna tem baixa seletividade (muitos valores iguais)
  • O planner tem informações ruins sobre o tamanho dos dados

Solução: Use ANALYZE para atualizar estatísticas: 

-- Atualiza estatísticas para que o planner tome melhores decisões
ANALYZE users;

-- Ou força um índice específico (último recurso!)
SELECT * FROM users 
WHERE country = 'Brazil' 
AND random_col > 0.5;

Estratégia 3: Particionamento

Para tabelas gigantescas, divida os dados em partições menores. Isso melhora performance e manutenção: 

-- Particionamento por range de data
CREATE TABLE logs (
    id BIGSERIAL,
    message TEXT,
    created_at TIMESTAMP
) PARTITION BY RANGE (DATE_TRUNC('month', created_at));

CREATE TABLE logs_2026_01 PARTITION OF logs
    FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');

CREATE TABLE logs_2026_02 PARTITION OF logs
    FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');

Dica final: Comece medindo. Use EXPLAIN ANALYZE. Procure pelos Seq Scan. Adicione índices estratégicos. Meça novamente. A otimização é iterativa, não é um evento único.
